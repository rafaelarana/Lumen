1 · Introduction

"One Postgres for data, vectors and keywords — no separate search cluster to run."

What is Lumen?

  • An e-commerce product search reference app over the WANDS catalog (~43K products).
  • Built entirely on Databricks: Lakehouse for data + embeddings, Lakebase (managed Postgres) for serving, a Databricks App (FastAPI + React) for the UI.
  • Two retrieval modes, chosen per request:
    • Semantic — vector similarity over BGE-large embeddings.
    • Hybrid — semantic + full-text, fused with Reciprocal Rank Fusion.
Everything — data, vector index, full-text index, fusion logic — lives in one database. Search is a SQL function call.

2 · Advantages vs traditional search engines

Fewer moving parts, governed, and good at both meaning and keywords.

Why Lumen, vs a dedicated engine

Operational simplicity

  • One system, not two — no separate search cluster.
  • No sync pipeline / dual source of truth.
  • Transactional consistency with the operational data.

Capability & governance

  • Semantic + keyword together, fused with RRF.
  • Governed by Unity Catalog; least-privilege grants.
  • Low latency via in-process caching (p99 ≈ 50 ms).
The trade: you give up a distributed engine's massive horizontal scale and BM25-grade text ranking — in exchange for one governed system with no synchronization. Great fit when search complements, rather than is, the product.

One database, not two

No separate cluster, no pipeline to keep in sync — search is a SQL function next to the data.

5 · How search works

Two signals, fused by rank — not by score.

Two signals

Semantic (vector)

  • Query → 1024-dim BGE-large embedding.
  • HNSW cosine search (<=>).
  • Captures intent & synonyms.

Full-text (keyword)

  • tsvector + GIN, plainto_tsquery.
  • Name-weighted ranking via setweight + ts_rank_cd.
  • pg_trgm fuzzy fallback for typos.
Postgres FTS has no true BM25/IDF — setweight + ts_rank_cd approximate it with field boosting + proximity.

How RRF fuses the two lists

Hybrid fusion — Reciprocal Rank Fusion

WITH vector_results AS (  -- top 3×k by cosine distance, ROW_NUMBER() = rk
  ... ORDER BY embedding <=> query_embedding ),
     text_results AS (    -- top 3×k by name-weighted ts_rank_cd, ROW_NUMBER() = rk
  ... ORDER BY ts_rank_cd('{0.1,0.3,0.6,1.0}', weighted_tsv, query) DESC )
SELECT product_id,
       0.7 * (1.0 / (60 + COALESCE(v.rk, 1000)))    -- vector weight
     + 0.3 * (1.0 / (60 + COALESCE(t.rk, 1000)))    -- text weight
       AS rrf_score
FROM vector_results v FULL OUTER JOIN text_results t USING (product_id)
ORDER BY rrf_score DESC LIMIT k;
Ranks are fused (not raw scores) because cosine distance and ts_rank_cd are not comparable. k=60 damping; missing rank → 1000.

How we measure performance

Search quality — "are results good?"

  • Ground truth: WANDS graded labels (Exact/Partial/Irrelevant).
  • Metrics: NDCG@k, Recall@k, Precision@k, MRR, MAP.
  • Harness in eval/ calls the SQL functions directly — no app, no cache → pure ranking.
  • Two query sets: clean + seeded-typo; reproducible (fixed seed & ef_search).

Latency — "is it fast?"

  • In-app load generator: async workers, 4 buckets (standard/turbo × semantic/hybrid).
  • Reports p50 / p95 / p99, req/s, error count per bucket.
  • External Goose load test adds Apps edge auth + WAN RTT.
  • Measured at each optimization stage to attribute the gain.
Quality and speed are measured separately and repeatably — so every change is judged on both, not on intuition.

Measuring latency — methodology

In-app benchmark (backend-only)

  • POST /api/benchmark spins async workers against localhost — no edge-auth noise.
  • Independently tracks 4 buckets: standard/turbo × semantic/hybrid.
  • Reports p50/p75/p95/p99/max, req/s, errors.

Goose load test (end-to-end)

  • Rust/Goose from a laptop → includes Apps edge auth + WAN RTT (~+60–100 ms).
  • Ramps users/concurrency to check scaling & error rate.
Same 100 seed queries in both; backend-only numbers isolate search cost, end-to-end numbers show what users feel.

6 · Comparison

Elasticsearch / OpenSearch & Vector DBs vs Lumen.

Lumen vs Elasticsearch / OpenSearch

DimensionElasticsearch / OpenSearchLumen (Postgres FTS + pgvector)
Text rankingBM25 (TF-IDF + length norm)ts_rank_cd + field weighting (no true IDF)
Text featuresfuzzy, synonyms, n-grams, highlighting, facetsstemming + pg_trgm fuzzy fallback
Scaledistributed, sharded, billions of docssingle-node (+ read replicas), 10⁴–10⁷ docs
Hybrid + vectorsnative (kNN + RRF retrievers, 8.x)RRF written in SQL; pgvector HNSW
Operationsseparate cluster + sync pipelineone Postgres; no sync; UC-governed
Same idea (inverted index + ranking + vectors); ES is the dedicated, distributed engine with better text ranking, Lumen folds it into the database.

Dedicated Vector DB vs Lumen

DimensionDedicated Vector DB (Pinecone, Weaviate, Milvus…)Lumen (pgvector in Lakebase)
Vector searchpurpose-built ANN, very large scaleHNSW via pgvector, ample for this corpus
Keyword / hybridoften add-on or externalfull-text + RRF in the same engine
Joins & filterslimited; metadata filters onlyfull SQL: joins, facets, transactional rows
Data movementseparate store to populate & syncnone — vectors sit next to the data
Governanceseparate IAMUnity Catalog + Postgres roles
When a dedicated engine wins: billions of vectors / very high QPS, or search-grade text features (BM25, synonyms, typo-tolerance everywhere) are the core product. Otherwise, keeping it in the database removes a whole system.
